# 0007 — Database Choice

**Status:** Accepted — 2026-02-28
**Deciders:** Team
**Technical Story:** Starport Registry persists starports, docking bays, customers, ships, reservations, routes, and outbox events. The database must support JSONB for flexible event payloads, transactional integrity for the Outbox Pattern, and schema migrations from day zero.

---

## Context and Problem Statement

Starport Registry (Service A) follows a Layered architecture backed by Spring Data JPA. The data model includes relational entities (starport, docking_bay, customer, ship, reservation, route) and the `event_outbox` table, which stores event payloads as JSONB. Which relational database engine best fits these requirements given the Spring Boot / JPA / Flyway stack and the local Docker Compose runtime?

---

## Decision Drivers

* JSONB column type required for `event_outbox.payload_json` and `event_outbox.headers_json` — allows Kafka message headers and payloads to be stored and indexed without schema churn.
* Full ACID transactions are mandatory: the Outbox Pattern relies on writing domain state and the outbox event atomically in one transaction.
* Flyway migration support must be first-class and tested against the real engine (not H2 compatibility mode).
* Spring Data JPA with Hibernate must work without surprises; `TIMESTAMPTZ`, sequences (`pooled-lo` optimiser), and `RETURNING id` must all function correctly.
* The engine must be runnable in Docker Compose as `postgres:16-alpine` for local development and Testcontainers for automated tests.
* No cloud-specific managed service should be required to run locally.

---

## Considered Options

1. **PostgreSQL 16 with Spring Data JPA + Flyway** — battle-tested, JSONB-native, rich type system.
2. **H2 in-memory (dev only)** — zero setup, but no JSONB, dialect differences, and unsupported DDL syntax (`GENERATED BY DEFAULT AS IDENTITY`, `TIMESTAMPTZ`).
3. **MySQL 8** — widely used, no native JSONB (uses `JSON` type with limited indexing), weaker support for `RETURNING` clause.
4. **MongoDB** — schemaless BSON documents, good for flexible payloads; but JPA does not apply, and relational integrity (reservations referencing docking bays) requires application-level enforcement.

---

## Decision Outcome

**Chosen option: PostgreSQL 16 with Spring Data JPA (Hibernate) and Flyway.**

PostgreSQL is the only option that satisfies all drivers simultaneously: native JSONB for the outbox, full ACID transactions, correct Flyway migration support, and a proven `postgres:16-alpine` Docker image for local use.

### Positive Consequences

* **JSONB for outbox payloads.** `event_outbox.payload_json JSONB` and `headers_json JSONB` allow rich, queryable event data without a separate schema per event type. The `ix_event_outbox_status_created` partial index supports the polling query efficiently.
* **Transactional outbox guarantee.** PostgreSQL's ACID isolation ensures that domain state and the outbox row are always committed together or not at all.
* **Reliable Flyway migrations.** All DDL in `V1__starport_basic_model.sql` and `V3__create_event_outbox.sql` uses PostgreSQL-specific syntax (`TIMESTAMPTZ`, `IDENTITY`, `JSONB`) tested against the real engine — no H2 dialect workarounds.
* **Sequence optimisation.** Hibernate `pooled-lo` optimiser (`hibernate.id.optimizer.pooled.preferred: pooled-lo`) aligns JVM sequence caching with database `INCREMENT BY 10` sequences, reducing round-trips.
* **Rich type system.** `TIMESTAMPTZ`, `NUMERIC(14,2)`, `DOUBLE PRECISION`, `TEXT` columns map naturally to Java types; no implicit coercions.
* **Testcontainers parity.** `PostgreSQLContainer("postgres:16-alpine")` in tests uses the identical engine and version as production — no dialect translation bugs.
* **`RETURNING id` support.** `jdbc.queryForObject("insert … returning id", Long.class)` in test fixtures works natively, simplifying acceptance test data setup.

### Negative Consequences

* **Extra infrastructure to run locally.** Docker is required; developers cannot run tests without a Docker daemon or without pre-starting the Testcontainers container.
* **JPA `open-in-view=false` discipline.** Lazy-loading pitfalls must be managed explicitly; `open-in-view` is disabled to prevent accidental N+1 queries from the web layer.
* **No application-level FK enforcement.** By design, foreign keys are intentionally omitted (see `V1` comments: `-- references starport.id (no FK by design)`) to allow test data setup via plain `INSERT` without insert-order dependencies. This trades referential integrity for test simplicity.
* **DDL-auto set to `none`.** Hibernate does not manage the schema; all DDL is owned by Flyway. Developers must write migrations for every schema change — no shortcut `ddl-auto=update`.
* **Postgres-specific DDL.** Migrations cannot be run against H2 or MySQL without changes.

---

## Pros and Cons of the Options

### Option 1 — PostgreSQL 16 + JPA + Flyway ✅

* Good, because native JSONB satisfies the outbox payload requirement.
* Good, because full ACID; transactional outbox is safe.
* Good, because Flyway + real engine in Testcontainers = zero dialect surprises.
* Good, because `postgres:16-alpine` is lightweight and fast to start.
* Bad, because requires Docker; no in-process fallback for pure-unit scenarios.

### Option 2 — H2 in-memory

* Good, because zero infrastructure; tests start instantly.
* Bad, because no JSONB support — outbox `payload_json` would require `TEXT` workaround.
* Bad, because `TIMESTAMPTZ`, `GENERATED BY DEFAULT AS IDENTITY`, and `RETURNING` are not supported in H2 PostgreSQL-compat mode without dialect patches.
* Bad, because migration differences hide real schema bugs until production.

### Option 3 — MySQL 8

* Good, because widely deployed; familiar to many teams.
* Bad, because `JSON` type lacks `@>` (containment), `->`, `->>` operators; outbox query optimisation is weaker.
* Bad, because `RETURNING` clause only from MySQL 8.0.21+ with `INSERT … RETURNING`; Spring/JDBC support is inconsistent.
* Bad, because Hibernate dialect differences require additional tuning.

### Option 4 — MongoDB

* Good, because BSON documents naturally model flexible event payloads.
* Bad, because Spring Data JPA does not apply — entire persistence layer would need Spring Data MongoDB.
* Bad, because multi-document ACID transactions (required for outbox) are only available in MongoDB with replica sets — complex to set up locally.
* Bad, because relational domain model (reservations ↔ docking bays ↔ starports) maps awkwardly to documents.

## Implementation

### Schema highlights

```sql
-- Outbox table (V3__create_event_outbox.sql)
CREATE TABLE event_outbox (
    id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    payload_json  JSONB,
    headers_json  JSONB,
    status        VARCHAR(16) NOT NULL
        CHECK (status IN ('PENDING','SENT','FAILED')),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ix_event_outbox_status_created ON event_outbox (status, created_at);
```

### Spring Boot configuration

```yaml
spring:
  datasource:
    url: ${DB_URL:jdbc:postgresql://localhost:5432/starports}
    driver-class-name: org.postgresql.Driver
  jpa:
    open-in-view: false
    hibernate.ddl-auto: none
    database-platform: org.hibernate.dialect.PostgreSQLDialect
    properties.hibernate:
      id.optimizer.pooled.preferred: pooled-lo
  flyway:
    enabled: true
    locations: classpath:db/migration
```

### Testcontainers

```java
@ServiceConnection
static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:16-alpine")
    .withDatabaseName("app").withUsername("test").withPassword("test");
```
